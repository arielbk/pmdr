## `native-close-button` — 2026-05-27 12:09:58

**Status:** needs-review
**Summary:** Swapped the floating panel from `[.borderless, .nonactivatingPanel]` to `[.titled, .closable, .fullSizeContentView, .nonactivatingPanel]` so macOS draws a real close traffic-light. `panel.titleVisibility = .hidden` and `panel.titlebarAppearsTransparent = true` keep the chrome invisible while letting AppKit handle the close button. Immediately after init the panel's frame is forced back to `Self.panelSize` via `panel.setFrame(NSRect(origin: .zero, size: panelSize), display: false)` because `.titled` inflates the frame by the title-bar height when interpreting the init `contentRect`. Miniaturize and zoom buttons are defensively `.isHidden = true`. The close button's `target` is set to `self` and `action` to `#selector(closeButtonClicked(_:))`, a new `@objc private func closeButtonClicked(_:)` that calls `hide()`. To make selectors and target/action work, `FloatingTimerPanelController` now inherits from `NSObject` and its init ends with `super.init()`. All other panel/contentView/effect-view construction is unchanged so the existing shadow-margin trick and hover wiring keep working.
**Deviations:** Marked `needs-review` rather than `done` because the slice has `Human checkpoint: yes` (eyeball that the close button matches native chrome and that content draws edge-to-edge with no title-bar gap), AND because this Linux sandbox has no Swift/AppKit toolchain so the structural tests can't be exercised here. One known visual risk: the standard close button is drawn in the title-bar slot at the top-left of the panel *frame*, which sits inside the 20pt shadow-margin band that currently surrounds the visual content. The button will therefore appear roughly 20pt above-and-left of the visible rounded rectangle. Two possible host-side fixes if the eyeball check confirms that: (1) drop `shadowMargin` and rely on `.titled`-derived shadow, retargeting the layer-corner rounding to the contentView; (2) reposition `closeButton` manually into the visual content top-left by editing its frame in its superview (the theme frame). I deliberately did not pick one — the human checkpoint is exactly the right place to choose. Structural review passed: `NSObject` superclass + `@MainActor` is a standard combination; `@objc private func` is reachable via `#selector` because the class is now NSObject-derived; `NSControl.target` is `weak`, so we don't create a retain cycle when the close button references `self`; `setFrame(_:display:)` immediately after `init` and before `panel.contentView = contentView` ensures the auto-resized contentView is sized to `panelSize`; existing position-persistence tests still record `panel.frame.origin` which is unaffected by styleMask changes.
**Handoff:** Added five tests to `FloatingTimerPanelControllerTests`: `testPanelChromeUsesTitledClosableAndFullSizeContentView` (styleMask flags + titleVisibility + titlebarAppearsTransparent), `testMiniaturizeAndZoomButtonsAreHidden` (nil-or-hidden tolerance because absent styleMask buttons return nil), `testCloseButtonTargetActionInvokesHide` (calls `target.perform(action, with: closeButton)` and asserts `panel.isVisible == false`), `testPanelFrameSizeIsUnchangedByTitledChrome` (`panel.frame.size == defaultPanelSize`), and `testCloseButtonRoundTripsPerMonitorPosition` (uses `TestScreen` + `FloatingTimerPosition`, sets origin, fires close via target/action, asserts the store records the position).
